1. 라우트는 가볍게

< 현재 초보자 시점에서 이러한 구조를 이해해야 하는 이유? 
처음 협업할 때 제일 많이 생기는 문제가 이거임
누군가는 route에 다 넣고
누군가는 service에 넣고
누군가는 model에 이상한 함수 넣고
누군가는 DB 코드까지 route에 쓸 수 있음.

그러면 팀 프로젝트는 결국 꼬여버림.
그래서 파일 역할을 먼저 통일해야 하기에 이렇게 모듈을 나눠 놓음.
==========================================================
< 해당 모듈에 대한 개념 설명 >
모듈 폴더가 많고, 파일이 많아서 좀 장황해 보일수 있지만...
실제로는 이 5개만 알면 이해가 그나마 쉬웠음.

1. route
역할 : 요청받기
비유 : 안내데스크 개념

( auth_routes.py )

2. service
역할 : 실제 처리 판단
비유 : 실무 담당자

( auth_service.py )

3. repositories
역할 : DB접근
비유 : 창고 담당

( member_repository.py )

4. models
역할 : DB 설계도
비유 : 서류 양식

( member_model.py )

5. schemas 
역할 : 요청 데이터 검증
비유: 신청서 검수 담당 (또는 접수 창구의 서류 검사)

6. common
역할 : 공통 도구
비유 : 공용 문서 / 도구함

( response.py )



==========================================================
< 이 모듈 구조에 대한 간략한 설명 >
run.py는 서버 실행 파일이고,
__init__.py는 Flask 앱을 조립하는 파일
config.py는 환경설정, extensions.py는 DB/JWT 같은 공통 확장 기능을 관리함.
routes는 요청을 받고, schemas는 입력값을 검사하고, services는 실제 로직을 처리함. 
repositories는 DB를 다루고, models는 테이블 구조를 정의함.
이렇게 나누면 협업할 때 역할이 명확해지고 유지보수가 쉬워지는 장점이 있음.


==========================================================
< run.py > 
모든 프로젝트의 실행 시작점!

가장 바깥쪽에서 서버를 실행하는 파일
이 파일은 말 그대로 프로그램 시작점!

< 역할 >
1. Flask 앱을 생성한다
2. 서버를 실행한다

< 핵심 코드 설명 >
from app import create_app

app 폴더 안에 있는 __init__.py 파일에서
create_app() 함수를 가져온다는 뜻임.

즉, “실제 Flask 앱은 app 폴더 안에서 만들고, 여기서는 불러와서 실행만 하겠다”

1. app = create_app()
Flask 앱 객체를 실제로 생성함
이 순간 내부적으로 설정 읽고 DB 연결 준비하고
JWT 연결하고 블루프린트 등록하는 과정이 진행됨
즉, 서버 뼈대를 만드는 핵심 함수 실행임.\


2. if __name__ == "__main__":
“이 파일을 직접 실행했을 때만 아래 코드를 실행하라” 는 뜻

예를 들어 python run.py 로 실행하면 아래 줄이 동작하고,
다른 파일에서 import만 하면 동작하지 않음.

3. app.run(debug=True)
Flask 개발 서버를 실행함
debug=True는 개발 중 오류를 화면에 자세히 보여주는 옵션 
특히 초보자 입장에서는..
에러가 났을 때 어디가 문제인지 더 잘 보이게 해주는 모드
